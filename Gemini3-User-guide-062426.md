食品藥物管理署 (FDA/TFDA) 審查人員專用操作與系統使用手冊
壹、 系統導論與開發背景
在新型醫療技術（如人工智慧軟體醫材 SaMD、數位療法的軟體裝置等）爆發式成長的時代，食品藥物管理署（FDA/TFDA）的醫療器材審查人員面臨著前所未有的挑戰。新興醫材不僅法規更新繁複，廠家提交的說明書、結構設計圖、臨床評估報告與品質管理系統（QMS）文件更是動輒高達數千頁。審查人員必須在極短時間內：
精確判讀並整理各類法規指引。
比對廠商送審目錄，篩選出「必要（Required）」與「非必要（Not Required）」文件，避免不必要的文件審查耗費行政資源。
撰寫格式嚴謹、用語客觀且具備法律與行政效力的「醫療器材審查報告」。
SmartMed Review 2.0 (智慧醫材審查指引與清單生成系統) 正是為了解決上述痛點而開發的次世代 AI 協作工具。本系統以 Google Gemini 3.1 Pro/Flash 先進大語言模型技術為核心，並特別構建了「六大 WOW 智慧亮點功能（WOW Features）」，旨在將審查人員從傳統、繁瑣的文字比對與報告堆砌中解放出來，讓其精力集中於更高價值的「核心臨床安全與功效評估」。
本手冊為 FDA 審查人員提供了全方位的系統模組操作指引、技術解析及系統調校設定說明。
貳、 系統介面基礎與通用控制
一、 介面版面配置 (Layout)
當審查員登入系統後，系統主畫面分為兩大主要區域：
左側固定式導覽列 (Navigation Panel)：包含系統商標、四個核心操作分頁（智慧指引生成、送審清單智慧分類、審查報告生成、系統設定），以及底部的「深淺色模式切換（Sun/Moon）」與「系統語系切換（繁中/英文）」。
右側主操作視窗 (Main Workspace)：根據所選分頁動態展示相關輸入、設定及即時生成的結果與控制台。
二、 深淺色模式與多語系支援
視覺友善設計：針對需要長時間閱讀高緊湊條文的審查員，點擊左下方 【淺色模式/深色模式】（或 Light Mode/Dark Mode）按鈕，即可瞬間轉換介面色溫。深色模式採用低對比度的鋅黑色調 (zinc-950)，能有效舒緩長時間用眼疲勞。
國際協同審查：點擊 【繁體中文/English】 按鈕即可即時切換整個系統的 UI 語系。當與國外監管機構（如 US FDA 或 PMDA）進行文件共閱或聯合審核時，可無縫切換為英文介面。

本模組協助審查員在面對未整理的外部新法規、草案或原廠技術規範時，能一鍵將其轉化為結構清晰、包含視覺化流程圖及法規溯源矩陣的「智慧審查指引」。

畫家風格 (Painter Persona) 下拉選單：為使原本刻板死沉的法規指引帶入不同的分析維度和寫作脈絡，本系統獨家整合了多位藝術巨匠的思考架構作為提示詞人格。
AI 模型下拉選單：審查員可自由選擇適合的 Gemini 引擎。
遙測控制台 (Telemetry Terminal)：提供底層系統連線、安全中止與生成時序的即時監控。
生成結果展示區：採用即時串流式（Streaming）Markdown 渲染技術，完整呈現分析報告。
步驟式操作指引
步驟 1：匯入法規條文
點擊畫面中的**【智慧指引生成】**分頁。
於**【法規依據輸入】**區塊中，可以選擇：
方法 A（檔案匯入）：拖曳相關的法規文件（如《行動醫療裝置檢驗準則草案.md》）置入虛線檔案上傳框，或點擊該區域手動選取檔案。
方法 B（文字複製）：直接將法規、指引草案的文字貼在「直接貼上文本」下方的多行文字框中。
步驟 2：配置畫家風格與 AI 模型
在下方**【畫家風格 (Persona)】**下拉選單中選擇合適的主題特徵（相關 Prompt 可至系統設定中編輯）：
預設審查員 (Default Reviewer)：專業、中立、嚴謹的科學立場。（初次審查推薦）
達文西 (Leonardo da Vinci)：著重人體功效學、機構工程精密比例與解剖結構的極致完美，適合外科植入式醫材與複雜構造評估。
梵谷 (Vincent van Gogh)：以宏觀、熱情、極具人文關懷與生命價值的視角演繹法規。
林布蘭 (Rembrandt)：擅長進行光影雙向解讀，會極力放大風險的「陰暗面」與效益的「光明面」，以利極端風險控管。
畢卡索 (Pablo Picasso)：從多重空間與非線性角度解構醫材設計，適合評估跨領域結合（例如 AI 結合微流體晶片）的新型智慧醫材。
達利 (Salvador Dali)：使用超現實主義隱喻及富有想像力的類比分析法規要求，常用於新創、顛覆性醫療技術的思維激盪。
於**【AI 模型】**選單選擇效能模型：
複雜、重邏輯的案件：推薦選擇 Gemini 3.1 Pro (Recommended for Complex Tasks)。
追求速度與快讀大文本：建議選擇 Gemini 3 Flash 或 Gemini 3.1 Flash Preview。
步驟 3：觸發生成與即時中止控制
點擊 【生成智慧指引】 按鈕。
此時，下方的 【遙測控制台 (Telemetry Terminal)】 將亮起，並即時輸出系統連線日誌，例如：
[XX:XX:XX] [SYSTEM] Initializing Guidance Generator...
[XX:XX:XX] [LLM] Connecting to gemini-3.1-pro-preview...
[XX:XX:XX] [LLM] Applying persona: 達文西...
[XX:XX:XX] [LLM] Receiving data stream...
若審查員發現貼錯法規內容或臨時有緊急修正，可於隨時點擊遙測終端機右下角的 【[STOP]】紅色按鈕。系統將透過底層 AbortController 機制中斷連線，安全停用當前生成，並記錄 [WARNING] User interrupted generation.，防止額外的運算與權限消耗。
步驟 4：閱讀、解讀與下載結果
指引輸出是採用 Stream 形式逐字渲染，當遙測日誌打出 [SUCCESS] Guidance generated successfully. 且載入動畫停止時，即代表生成完成。
生成的 Markdown 將動態解析為符合 TFDA 審查標準之結構：

【WOW 3】 模擬退件風險評估：預測廠商若遺漏各項核心資料時的被退件機率，例如：「因未提供出產國許可製售證明（CFS）可能導致高達 90% 的直接退件風險」。

【WOW 4】 法規溯源熱區圖 (Regulatory Traceability Heatmap)：呈現合規對照表格，讓審查員一目了然各項查核點背後的法源。
點擊生成結果右上角的 【下載下載圖示 (Download)】，系統會將整篇 Markdown 生成一份名為 guidance.md 的檔案供下載離線存檔。

廠商送審清單輸入面板 (Submission List)：可用於直接貼上或上傳（支援 .txt, .csv, .md, .json 等格式）廠商所附的原始文件目錄與項目摘要。

智慧分類結果展示板：將龐大的清單轉化為紅（必要）、灰（非必要）、藍（選用）三個極具視覺指引力的獨立卡片卡槽，各項卡片詳列對應法源依據與判定理由。
步驟式操作指引
步驟 1：匯入廠商檢附清單
切換至 【送審清單智慧分類】 分頁。
於左上方的 【廠商送審清單】 區塊，點擊上傳按鈕匯入廠商提供的送審清單檔案（例如 廠商送審清單_AI腦影像助理軟體.txt），或直接於下方的文字框貼入清單內容。
步驟 2：校閱並導入比對指引
在左下方的 【審查指引/法規依據】 區塊內，預設已填寫常見核發與展延法語。
若您承辦的案件屬於「特定產品別（如：第二等級體外診斷試劑、人工心肺系統技術基準）」，請在此欄位貼入該品項的官方特定技術審查指引。AI 將會以此指引為黃金標準（Golden Standard）對其進行嚴格比對。
步驟 3：選擇模型並執行分類
確認 AI 模型 設定（本功能為精準結構化 Schema 輸出，需要正確判定 Required/Optional，強烈建議使用 Gemini 3.1 Pro 進行高維語意抽取）。
點擊 【執行智慧分類】。
透過遙測終端機監控底層連線狀態。
步驟 4：分析分類結果與導出
生成完畢後，右側將清晰呈現三大卡槽：
必要項目 (Required Items)：以紅色高亮警示。列出該案必備的文件（例如「醫療器材品質管理系統證明文件 (QMS)」、「臨床試驗驗證資料」等），並詳細列出法規溯源條文（如「核發準則第17條」或「第6條」）以及若遺漏此項目的退件風險原因。
非必要項目 (Not Required Items)：以灰色呈現。智慧識別出依法得予豁免的項目（例如第一等級產品依法免核定成分規格、或專供外銷醫材得免部分文件），節省審查員對非必要項目的審驗力氣。
選用/建議項目 (Optional Items)：以藍色高亮呈現。列出非強制，但若檢附將大大加速安全功效評估的佐證材料（例如：他國已核准上市 CFS 證明、國際第三方臨床試驗報告），並標示其參考與推薦原因。
點擊右上方的 【匯出 JSON (Export JSON)】 按鈕，系統將當前結構化分析好的分類結果打包為法規合規標準的 .json 檔案。此檔案可用於登錄局處內部資料庫，或提供廠商作為「文件缺件核對與改善備忘錄」。

三大資料來源輸入框：
廠商送審摘要：提供上傳與貼上功能，放至主要技術重點摘要。

編輯與預覽雙模式沙盒(Edit/Preview ViewMode)：生成出來的報告，可以隨時在 「唯讀 Markdown 預覽模式」、以及 「直接可修改之純文字 Raw Text 編輯模式」 間切換，供審查人員手動精微調改核心措辭。
步驟式操作指引
步驟 1：輸入廠商技術摘要
進入 【審查報告生成】 頁面。
於第一個核心區域貼入（或上傳）此案的技術大綱、廠商基本背景、產品結構與材質宣稱等資訊。
步驟 2：確認引用法規與審查模板
確認「審查指引」是否已貼入正確的對應條款與技術規範。
「報告模板」已被預先填入多維度的標準 Markdown 框架。您可以選擇直接點擊模板右上角的 【下載模板 (Download)】 以保存，亦可在文字編輯區內微調模板結構以符合個別處室的簽呈規範。
步驟 3：模型派發與生成
選擇模型（Gemini 3.1 Pro 針對此 3000~4000 字之長文本，其上下文窗 (Context Window) 能完美控管報告前後文的一致性與法行政用語品質）。
點擊 【生成審查報告】，並於 Telemetry Terminal 中監管寫作進度。
步驟 4：交互沙盒編輯與補件問答
報告產出後，系統預設進入 【預覽模式 (Preview)】，審查員可以看見完美的 Markdown 格式呈現，以及末端附贈的兩大亮點功能：
【WOW 5】 自動生成補件問答集 (Auto-Generated Clarification Q&A)：AI 會就廠商文件中含糊不清、或前述被歸類為「必要項目卻可能有缺陷」的部分，精準產出 3 到 5 道語意犀利、合規的追問問題。例如：「請說明本產品在出廠滅菌流程（γ-Sterilization）中的確效（ValidationError）係數，並補足相應出產國實體管制證明。」此問答集可直接拷貝，放入 TFDA 通知補件之官方文書。
【WOW 6】 審查報告語氣與合規性分析 (Tone & Compliance Analyzer)：AI 將在報告底部，對本篇產出的審查報告進行「客觀性、科學嚴謹度、法效字眼合規度」的智能分析報告，確認該語氣中性溫和，不偏頗任何單一商業技術。
切換 【編輯 (Edit Mode)】 按鈕，右側展板立即轉為可點選修改的文字編輯器。審查員可直接對報告內容進行人工覆核、加註局處名稱、修改承辦人等微調。
點擊 【下載 MD (Download MD)】 下載生成的審查報告電子檔供內部簽核使用。

編修完成後點擊右上角 【儲存變更 (Save Changes)】 綠色按鈕。保存後，整個系統各個分頁在啟動 Gemini 動態串流與分類時，將自動套用您新定義的監管邏輯提示。
柒、 「六大 WOW 功能」應用實務與原理精解
WOW 亮點功能	名稱	技術原理與運作機制	審查員核心應用場景
WOW 1	Mermaid 視覺化審查流程圖	基於 Markdown 內嵌的 MermaidJS 圖形編譯器，自動分析法規流程邏輯，輸出 Class I/II/III 審查路徑與查點節點圖。	用於主管及外行人員快速理解此類產品在 TFDA 常規審查與特別登錄（優先審查）的路徑差異。
WOW 2	國際法規智能對照	借助 LLM 本地對跨監管（FDA/EMA/TFDA）資料庫的語意理解，完成 Product Code 對照及安全分等比對。	當廠商送審文件主要是美規 FDA 510(k) 或歐盟 CE 時，可以瞬間定位其對應 TFDA 之本土定位與風險級等。
WOW 3	模擬退件風險評估	風險係數推演引擎。基於以往海量退件或補件歷史知識，估計缺失項目（如缺乏實體 CFS 或 QMS 生產確效書）之退件率與補件機率。	起草核准意向書或決意書前，對缺失項目提供科學、量化的警告。
WOW 4	法規溯源熱區圖	結構化語意檢索表格。將審查報告大綱中抽取的查核點，精準對應到《醫療器材管理法》及關聯準則條款。	當面對廠商對法律合規性提出疑慮時，本熱區表可作為不可動搖、白紙黑字的執法條文論理依據。
WOW 5	自動生成補件問答集	發散式合規提問生成算法。專門針對「必要項目缺失」與「選用項目不足」，自動構建符合監管官方語氣之澄清提問。	審查員可直接拷貝該問答集發送給廠商，大幅省卻撰寫各類複雜、繁瑣、合規之補件公文的时间。
WOW 6	報告語氣與合規分析	文本偏見與科學一致性查驗。在送出報告前，自我進行二次語意合規性審視，確保無過多主觀色彩或情緒性字眼。	作為 TFDA 報告最後的「智能文字品管與防腐機制」，降低後續面臨廠商訴願、訴訟時的語意漏洞風險。
捌、 系統故障排除與常見問答 (Troubleshooting)
Q1：點擊生成按鈕後出現 [ERROR] 連線失敗，或者沒有反應？
排除方式：請點擊右上角並檢視您的網路連接。若確認網際網路通暢，則可能為 API 金鑰設定缺失。本系統之 Google Gemini API Key 由平台底層環境或設定面板自動注入；請確認環境變數 GEMINI_API_KEY 有效。若仍無法排除，可至「系統設定」微調預設模型，並嘗試重新點擊生成。
Q2：Mermaid 流程圖顯示為一堆雜亂文字，沒有畫出美麗的圓框與組織線？
排除方式：Mermaid 編譯器高度依赖語法正確性。本系統已在 Prompt 中寫入嚴格防錯基準：「注意：Mermaid 節點文字若包含括號或特殊符號，請務必使用雙引號包覆」。若偶爾仍發生轉譯失敗，在 【審查報告生成】 的 【編輯模式】 中，將寫有 mermaid 代碼內文字中含有的特定特殊字元（如中括號、圓括號、半形標點）補上雙引號（例如：A["Node 1 (Class II)"]），再切換回預覽，即可成功繪製。
Q3：如何中止持續太久的生成？
排除方式：各個頁面均配備專屬的 【遙測控制台 (Telemetry Terminal)】。該控制台具備中斷主控元件。只要點擊控制台內的 【[STOP]】，便可透過呼叫原生的 AbortSignal 來攔截與中止 API 連線，立刻釋放資源，不用重洗頁面。
